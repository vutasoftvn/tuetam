# Kids Coloring Next.js Design

## Goal

Build a Next.js web app for children aged 3 to 6 to color drawings on a tablet. The app should feel simple, bright, and touch-first, with many templates around flowers, leaves, fruits, animals, and toys. Coloring must stay inside closed regions, so children get a clean result even with broad finger movements.

## Confirmed Decisions

- Primary device: tablet/iPad in landscape orientation.
- Main layout: template library on the left, large coloring canvas on the right.
- Coloring behavior: fill closed regions only; color must not bleed outside outlines.
- Brush sizes: two choices, `Vừa` and `To`.
- Template source: built-in library plus parent-uploaded templates.
- Persistence: save locally in the browser without login.
- Reset: each picture can be reset and colored again.
- Colors: each template has suggested colors, but children can choose freely from those suggestions.

## User Experience

The first screen is the coloring studio, not a landing page. The left side shows template categories and thumbnails. The right side shows the active drawing at large size. The bottom toolbar contains large, touch-friendly controls for colors, brush size, undo, reset, and save status.

Children should need very little reading. Controls should use icons, color swatches, and short labels only where necessary. The interface should avoid nested menus in the child-facing studio.

The left template library includes these categories:

- Hoa
- Lá
- Quả
- Con vật
- Đồ chơi
- Mẫu của ba mẹ

The parent upload area is separate from the child studio. A parent button opens a panel for adding and managing custom templates. This keeps the child-facing interface calm and simple.

## Core Screens

### Coloring Studio

The studio contains:

- Left template library with category tabs and large thumbnails.
- Main coloring surface.
- Suggested color palette for the active template.
- Brush size segmented control: `Vừa` and `To`.
- Undo button.
- Reset button for the active picture.
- Saved status indicator.

### Parent Template Panel

The parent panel contains:

- Upload control for SVG and PNG line-art templates.
- Validation result for uploaded files.
- Preview of accepted templates.
- Delete option for parent-added templates.
- Guidance when a file cannot support region coloring.

The MVP supports SVG and PNG uploads with different quality expectations. SVG is preferred because paths can carry stable region identifiers. PNG line-art is accepted when the image has clear dark outlines and mostly closed regions; the app uses bitmap flood fill bounded by the detected line art. If a PNG has gaps, noisy backgrounds, or low contrast, the app warns the parent that region coloring may not be accurate.

## Coloring Model

Built-in templates should be normalized SVG line-art. Each colorable region is a closed SVG shape or path with a stable `regionId`, such as `petal_1`, `leaf_2`, or `wheel_1`.

Parent-uploaded templates can use two coloring models:

- SVG: store region fills by stable SVG region ID.
- PNG: store a raster fill layer generated by flood fill. The original PNG remains unchanged underneath.

The app stores coloring progress as structured data:

```ts
type ColoringProgress = {
  templateId: string;
  fills?: Record<string, string>;
  rasterFillDataUrl?: string;
  updatedAt: string;
};
```

For example:

```json
{
  "templateId": "flower-01",
  "fills": {
    "petal_1": "#F87171",
    "leaf_1": "#4ADE80"
  },
  "updatedAt": "2026-06-10T00:00:00.000Z"
}
```

This keeps saved data small and makes reset simple: remove the progress object for the current `templateId`.

For PNG uploads, saved data is larger because it includes the raster fill layer. This is acceptable for local browser storage, but the app should limit uploaded image dimensions to protect performance on tablets.

## Brush Behavior

Although the app fills regions rather than drawing freehand pixels, the brush size still matters for touch interaction:

- `Vừa`: smaller hit radius, better for choosing nearby small regions.
- `To`: larger hit radius, easier for young children and larger regions.

When a child taps or drags, the app detects colorable regions under the touch path. The selected color is applied to those regions only.

## Template Data

A built-in template should include:

```ts
type ColoringTemplate = {
  id: string;
  title: string;
  category: "hoa" | "la" | "qua" | "con-vat" | "do-choi" | "parent";
  sourceType: "svg" | "png";
  thumbnail: string;
  svg?: string;
  imageDataUrl?: string;
  suggestedColors: string[];
  regionIds?: string[];
};
```

Built-in examples should cover at least:

- Flowers: simple flower, sunflower, tulip, lotus.
- Leaves: single leaf, branch with leaves, autumn leaves.
- Fruits: apple, banana, orange, strawberry, watermelon.
- Animals: cat, dog, fish, bird, butterfly.
- Toys: teddy bear, ball, kite, car, blocks.

## Upload Validation

Uploaded templates should be accepted only when they can support child-friendly region coloring.

Validation checks:

- File type is SVG or PNG.
- File size is under the configured limit.
- SVG contains colorable closed paths or shapes.
- SVG colorable regions can receive stable IDs.
- SVG does not contain unsafe script content.
- PNG has sufficient contrast between outline and background.
- PNG dimensions are within the configured tablet-friendly limit.

If validation fails, the app shows a short parent-facing reason. Examples:

- `File này chưa có vùng tô rõ ràng.`
- `Một số nét chưa khép kín nên màu có thể bị lem.`
- `File quá phức tạp cho phiên bản hiện tại.`
- `Ảnh PNG cần nền sáng và nét viền rõ hơn.`

## Local Storage

Use IndexedDB for:

- Coloring progress for built-in and parent templates.
- Parent-uploaded template metadata and SVG/PNG content.
- Basic app preferences such as last selected brush size.

No account or server persistence is required for the first version.

## Next.js Architecture

Use Next.js App Router. The drawing experience should be implemented as client components because it depends on browser touch events and SVG interaction.

Suggested structure:

```txt
app/
  page.tsx
components/
  coloring-canvas/
  parent-panel/
  template-library/
  tool-bar/
lib/
  png/
  storage/
  templates/
  svg/
```

Responsibilities:

- `app/page.tsx`: page shell and studio layout.
- `components/template-library`: categories, thumbnails, active template selection.
- `components/coloring-canvas`: SVG rendering, touch hit detection, region fill application.
- `components/tool-bar`: color swatches, brush size, undo, reset, save status.
- `components/parent-panel`: upload and parent template management.
- `lib/templates`: built-in template registry.
- `lib/storage`: IndexedDB access and persistence helpers.
- `lib/svg`: SVG sanitization, validation, and region normalization.
- `lib/png`: PNG validation, outline thresholding, and flood-fill helpers.

## Error Handling

Child-facing errors should be rare and gentle. Parent-facing upload errors can be more specific.

Expected cases:

- Storage unavailable: continue in memory and show a small parent-facing warning.
- Uploaded SVG invalid: reject with a clear reason.
- Uploaded PNG invalid or too low contrast: reject with a clear reason.
- Template fails to load: show a fallback empty state and keep the library usable.
- Reset confirmation: use a simple confirmation because reset removes current coloring progress.

## Testing Plan

Test coverage should focus on behavior that protects the core experience:

- Selecting a template loads the correct SVG and suggested palette.
- Tapping a region applies color only to that region.
- Dragging with `Vừa` and `To` affects the expected regions.
- Changing color and recoloring a region updates the saved fill.
- Progress persists after page reload.
- Reset removes only the active template progress.
- Valid SVG uploads are accepted into `Mẫu của ba mẹ`.
- Valid PNG line-art uploads are accepted into `Mẫu của ba mẹ`.
- Invalid or unsafe SVG uploads are rejected.
- Low-quality PNG uploads are rejected or warned before use.
- Layout remains usable on iPad landscape dimensions.

## Out Of Scope For First Version

- User accounts.
- Cloud sync.
- Freehand pixel drawing.
- Advanced PNG cleanup for noisy photos or open-line sketches.
- Stickers, rewards, lessons, or gamified missions.
- Printing workflow beyond standard browser save/download options.
